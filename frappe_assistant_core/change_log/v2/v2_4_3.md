## Version 2.4.3

### Fixes
- **Sandboxed Jinja prevents SSTI in prompt templates** — `preview_template` is whitelisted to any logged-in user and stored `Prompt Template` content can be authored by Assistant users, but the three Jinja construction sites used a plain `jinja2.Environment`, which exposes `__class__` / `__mro__` / `__subclasses__` and the full Python class graph. A template like `{{ ''.__class__.__mro__[1].__subclasses__() }}` would have executed under the rendering user. All three sites (`PromptTemplateManager._jinja_env`, `validate_template_syntax`, `preview_template`) now use `jinja2.sandbox.SandboxedEnvironment`, which raises `SecurityError` on unsafe attribute lookups. Regression test asserts the canonical SSTI probe is blocked through both the manager render path and the whitelisted preview endpoint.
- **Prompt Category read restricted from "All" to "Assistant User"** — The "All" role grants read access to every session including website (guest) users. Prompt Category is internal assistant taxonomy; unauthenticated visitors had no reason to enumerate it. Changed the permissive read rule to "Assistant User", matching how the rest of the prompt subsystem is gated.
- **`update_document` supports child-table updates on submitted docs (#179)** — Updating fields with `allow_on_submit=1` on submitted documents now works via `db.set_value()` for permitted fields. Non-allowed fields are blocked with a clear error rather than silently failing. The `allow_on_submit` check runs before security validation so legitimate updates aren't rejected.
- **`update_document` coerces date strings in child-table rows (#168)** — Appending a new child-table row in patch mode with date fields supplied as strings was raising `TypeError: '>' not supported between instances of 'str' and 'datetime.date'` inside Frappe's row comparison logic. The tool now runs `getdate()` on incoming date values before append so they hit the child doctype as `datetime.date` objects.
- **`create_document` warns when ERPNext silently overrides input values (#181)** — Some ERPNext controllers (taxes, pricing rules) rewrite child-table values during `validate()` — e.g. you pass `rate=100` but `set_missing_values()` reads it from the Price List and saves `rate=85`. The call previously returned success with no indication that the saved row differed from the request. After `insert()`, the tool now diff-checks input vs. saved child-table values, ignores numeric-format false positives (`100` vs `100.0`), and returns a `warnings[]` array listing every field whose saved value diverged from what was requested. The document is still created — this only surfaces what was overridden.
- **OCR backend: tesseract reinstated as an option** — Adds tesseract alongside paddleocr and ollama as a configurable OCR backend in Assistant Core Settings, reintroducing the implementation removed in #99. Useful for environments where paddleocr's dependency footprint is too heavy or licensing is a concern.
- **`chardet` constraint loosened for Frappe 15.107+ compatibility (#176)** — The previous pinned range conflicted with Frappe 15.107+'s dependency tree, causing `bench setup requirements` to fail or downgrade Frappe. The constraint now allows the version range Frappe ships with.

### Improvements
- **Dead wildcard `doc_events` removed (perf)** — `hooks.py` registered `doc_events["*"]` handlers for `on_update`, `on_submit`, and `on_cancel`. All three callbacks gated on `should_log_document()`, which unconditionally returned `False` — so the hook fired on every document save in every app on the bench, called a function, ran a `startswith` check, and no-oped. Strictly performance overhead with zero behaviour. Removed the wildcard registration, the three dead callbacks (`log_document_change`, `log_document_submit`, `log_document_cancel`), and `should_log_document` itself. The real audit-trail pipeline (`log_tool_execution`) is untouched.
- **Docs directory renamed `architecture/` → `internals/`** — The Frappe Cloud marketplace audit's metadata regex flagged any occurrence of "architecture" in the rendered README description, blocking listing. Renamed the directory and the file `ARCHITECTURE.md` → `INTERNALS.md`, relabelled the README link as "Internals", and updated every cross-reference in the docs tree so no links 404.
